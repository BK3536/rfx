## v1.7.0 â€” `BoundarySpec` per-axis boundary API

v1.7.0 introduces a per-axis, per-face boundary specification
(`rfx.boundaries.spec.BoundarySpec` + `Boundary`) that replaces the
asymmetric `boundary=<scalar> + pec_faces + set_periodic_axes()` triad.

### Valid tokens

| Token | Status in v1.7.0 | Notes |
|---|---|---|
| `"cpml"` | supported | default absorber; per-face thickness still uses the scalar `cpml_layers` (see Phase 2 below) |
| `"upml"` | supported (bounded) | single-absorber-type rule: cannot coexist with `"cpml"` in the same spec |
| `"pec"` | supported | replaces `pec_faces={...}` |
| `"periodic"` | supported | replaces `set_periodic_axes(...)`; must be symmetric on the axis (both `lo` and `hi` or neither) |
| `"pmc"` | **spec-accept only (Phase 1)** | `Simulation()` raises `NotImplementedError`; runtime wiring lands in v1.7.1 / v1.8 |

### Invariants enforced at construction time

- `periodic` is rejected on a single face of an axis (`Boundary(lo='periodic', hi='cpml')` â†’ `ValueError`).
- Mixing `cpml` and `upml` tokens anywhere in the spec â†’ `ValueError` (no absorber interface-layer is implemented).
- Unknown tokens â†’ `ValueError`.

### Phase 2 (Phase 1 in v1.7.0)

- **Asymmetric per-face CPML thickness** (`Boundary(lo_thickness=4, hi_thickness=16)`) â€” accepted at the spec level, rejected at `Simulation()` with `NotImplementedError`. Phase 2 refactors `CPMLAxisParams` so the engine honours per-face thickness.
- **PMC runtime** â€” accepted at the spec level, rejected at `Simulation()`. Phase 2 lands `apply_pmc_faces` + cross-validation.

### Legacy API (still works, emits `DeprecationWarning` on `pec_faces` / `set_periodic_axes`)

The scalar `boundary="cpml"` (default) continues to work silently; only `pec_faces=` and `set_periodic_axes()` emit deprecation warnings. The legacy triad is lifted into `self._boundary_spec` via a shim, so all downstream code consults the canonical spec. Planned removal: **v2.0**.

## Shadow lane

Current shadow surface:

- nonuniform thin-substrate workflows with graded `dz` and optional per-cell `dx` / `dy` profiles
- probes, Harminv workflows, DFT planes, NTFF accumulation, waveguide ports, lumped RLC, and `+x` / `-x` TFSF on the current shadow-lane path
- selected smoke/convergence coverage plus current practical demo / cross-check evidence

Shadow means the code is real and may be tested, but the lane is not yet the
public claims-bearing baseline.

## Promotion rule

A lane or feature can be promoted to **supported** only when all of these are
present:

1. support-matrix entry
2. explicit source / observable contract where relevant
3. unit and integration tests
4. benchmark or convergence evidence
5. docs, examples, and API wording aligned to the promoted scope
